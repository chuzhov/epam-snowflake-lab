EPAM Snowflake Hands-on Lab— Project Summary

A hands-on coverage of Snowflake's core and advanced feature set — Snowpipe, append-only Streams, scheduled and stream-gated Tasks, Snowflake Scripting stored procedures, Dynamic Tables, External Tables, Time Travel, Zero-copy Cloning, and Secure Data Sharing — grounded in a realistic CRM export workflow: a nightly Azure Function mimicking a CRM's daily order export drops fresh files into ADLS Gen2, three parallel ingestion paths process the data, and a star schema data mart serves Power BI.

💡 Key Design Decisions

Why CSV (not Parquet)?

Matches real CRM export format. The lab brief specified external stage + file format handling, which is more instructive with CSV — header skip, quote handling, and explicit date format all have to be configured.

Why pre-upload all 15 daily pairs to seed/future_batches/?

splitter.py shifts all TPC-H dates so the max o_orderdate lands exactly 15 days from the run date. Pre-generating gives a deterministic, repeatable dataset without running the generator again each day.

Why Event Grid → Queue Storage instead of Snowpipe REST insertFiles?

Decoupled architecture: Snowpipe auto-retries, no Snowflake credentials live in the Azure Function, and the pattern is standard in enterprise pipelines. The REST API alternative tightly couples crm_mimic to Snowflake.

Why the ADMIN schema with persistent log tables?

SYSTEM$PIPE_STATUS() returns live state only; COPY_HISTORY is a transient view with 14-day retention. Persisting both gives crm_archival a durable, queryable audit store — blob lifecycle decisions are driven by data, not timing assumptions.

Why parallel schemas instead of replacing existing ones?

The Basic lab pipeline had 14 M+ historical rows accumulated and was graded. Replacing CORE_DWH in place would require a full re-backfill. Parallel schemas (CORE_DWH_EXT, DATA_MART_DT) demonstrate the contrast directly with zero risk.

Why INSERT_ONLY (not APPEND_ONLY) on external table streams?

Snowflake requires INSERT_ONLY = TRUE for streams on external tables — it tracks file additions, not row-level changes. APPEND_ONLY (used on regular-table streams in STAGE_AREA) is not valid here and will be rejected at DDL time.

Why Dynamic Tables for DATA_MART_DT?

DT_FACT_LINEITEM is a 20-line SELECT vs. the ~50-line imperative procedure in sp_core_to_mart. No stream management, no task wiring, no DML. One constraint: Dynamic Tables do not support AUTOINCREMENT — the natural TPC-H composite key (l_orderkey, l_linenumber) is used instead of a surrogate.

☁️ Cross-Cloud Implementation Notes (Azure storage + AWS-hosted Snowflake)

The Azure subscription was provisioned first; the Snowflake trial account was then created on AWS (Snowflake's default trial provider) rather than on Azure. The pipeline therefore runs cross-cloud — storage and event services in Azure, the warehouse in Snowflake-on-AWS. This is a realistic multi-cloud setup, but it removed three event-driven primitives the lab brief assumes are available same-cloud. Each was replaced with a scheduled-task or log-scanning equivalent:

Cross-cloud limitationSame-cloud "happy path"Workaround used here
External tables can't AUTO_REFRESH — event-driven auto-refresh needs storage and Snowflake to share a cloud notification channel; unsupported when ADLS (Azure) feeds Snowflake-on-AWS AUTO_REFRESH = TRUE + INTEGRATION on the external table task_refresh_ext_tables — scheduled task (18:05 UTC) issuing ALTER EXTERNAL TABLE … REFRESH on both external tables
Snowpipe status not directly usable by the archival step — SYSTEM$PIPE_STATUS() returns live state only, isn't queryable from Azure, and cross-cloud notification latency made timing-based archival unreliable Trust auto-ingest timing; check pipe status inline The crm_archival Azure Function authenticates into Snowflake (RSA key-pair) and scans the persistent ADMIN.PIPE_STATUS_LOG / PIPE_INGESTION_LOG tables (populated by sp_refresh_pipe_log, 18:30 UTC) to drive blob archival from data, not timing
Extra Azure hop for Snowpipe notifications — AWS S3 emits events to SQS natively; Azure has no direct Blob→queue path S3 event → SQS → Snowpipe poll Event Grid (BlobCreated on raw/) → snowpipe-notifications Storage Queue → Snowpipe polls the queue

Net effect: every place the brief assumed a same-cloud event-driven primitive, it was replaced with a scheduled-task or persistent-log equivalent — which also made the pipeline more observable and auditable than the event-driven default would have been.

🧪 SQL Tests Performed in Snowflake

Row Count Validation — Phases 3 & 5

After bulk COPY INTO, a UNION ALL across all 8 STAGE_AREA tables confirmed TPC-H SF=2 counts match the official spec exactly (ORDERS = 2,981,311 · LINEITEM = 11,920,598 · CUSTOMER = 150,000 · PART = 200,000 · PARTSUPP = 800,000 · SUPPLIER = 10,000 · NATION = 25 · REGION = 5). After the ELT chain ran, CORE_DWH and DATA_MART row counts were compared against STAGE_AREA to confirm all rows were promoted without loss or duplication.

Snowpipe Health & Ingestion Audit — Phase 4

Both pipes validated after each daily run using three queries in combination: SYSTEM$PIPE_STATUS() (confirms executionState = RUNNING and pendingFileCount), INFORMATION_SCHEMA.COPY_HISTORY() (rows ingested and errors per file over the last 24 h), and the persistent ADMIN.PIPE_STATUS_LOG / PIPE_INGESTION_LOG tables populated by sp_refresh_pipe_log. Task completion verified via INFORMATION_SCHEMA.TASK_HISTORY().

TPC-H Warehouse Size Benchmark — Phase 7 (Task 5)

All 22 standard TPC-H queries run with USE_CACHED_RESULT = FALSE across three warehouse sizes in sequence — X-SMALL → SMALL → MEDIUM — producing 66 total executions tagged tpch_bench. Results extracted from INFORMATION_SCHEMA.QUERY_HISTORY and pivoted to compare elapsed_ms, compile_ms vs exec_ms, mb_scanned, and XS→SM / SM→MD speedup ratios per query.

Object Cloning — Phase 8 (Task 6)

Zero-copy clone of the entire CORE_DWH schema created instantaneously (CREATE SCHEMA … CLONE). A destructive migration (setting o_orderstatus = 'S' for old open orders) applied to CORE_DWH_CLONE.ORDERS was confirmed to leave production untouched. Clone dropped after validation — no extra storage billed while data was shared with the original.

Time Travel Recovery — Phase 8 (Task 6)

Simulated corrupt batch ingestion (1,000 rows with o_orderdate shifted +1 year). Two recovery paths tested: (1) surgical DELETE (WHERE o_comment LIKE 'CORRUPT BATCH%') and (2) full restore via Time Travel clone (CLONE STAGE_AREA.ORDERS AT (OFFSET => -300), anchored 5 minutes before the incident). Post-recovery: COUNT(*) WHERE o_orderdate > CURRENT_DATE = 0.

Secure Data Sharing — Phase 8 (Task 6)

Named share created exposing all DATA_MART tables (5 dims + fact) with no data copying: CREATE SHARE crm_datamart_share + GRANT SELECT ON ALL TABLES IN SCHEMA EPAM_LAB.DATA_MART. Also provisioned a managed Reader Account (finance_reader_acct) for consumers without their own Snowflake subscription — consumer queries execute against the provider's micro-partitions, always live, zero ETL.
